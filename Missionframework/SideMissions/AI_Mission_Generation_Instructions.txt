AI Mission Generation Instructions (Gold Standard v2.2, Community Range _4–_99)

Important blind-test rule:
When running in a web/chat environment without trusted folder visibility, the AI must treat ID availability as unknown.
It must ask for confirmation before using _4 or any other community ID.

Purpose
These instructions apply only to community-authored side mission templates in the _4–_99 range.
Official/internal missions _1–_3 are reserved and may be used as style references, but must not be overwritten,
duplicated, renamed, or copied into the community range without assigning a new ID.

Current runtime state
- land side missions are active
- naval side missions are not active yet
- air side missions are not active yet
- community AI mission generation should target land only until naval and air are explicitly enabled later

1. Real folder structure (MANDATORY)
Use the real OIM folder structure exactly as it exists now:

Missionframework/SideMissions/land/<category>/<difficulty>/<file>.sqf

Allowed land category folders:
- supply
- intel
- disrupt

Allowed difficulty folders:
- easy
- medium
- hard

Valid filename prefixes:
- MissionSupply_<id>.sqf
- MissionIntel_<id>.sqf
- MissionDisrupt_<id>.sqf

Examples:
- Missionframework/SideMissions/land/supply/easy/MissionSupply_4.sqf
- Missionframework/SideMissions/land/intel/medium/MissionIntel_7.sqf
- Missionframework/SideMissions/land/disrupt/hard/MissionDisrupt_12.sqf

Do not use the old deprecated structure:
- SideMissions/modern/...
- SideMissions/DisruptMission/...
- SideMissions/MainOperation/...

Those paths are not the current land side mission format.

2. Reserved vs community mission IDs
- _1–_3 = official/internal reserved range
- _4–_99 = community AI mission range

Community AI generation must never modify _1–_3.
Use _1–_3 only as reference examples for style, structure, and field order.

3. ID selection rules (MANDATORY)
Before creating a new community mission, the AI must first determine which numeric IDs already exist in the exact target folder.

Normal workflow:
1. inspect the exact target folder
2. list all used IDs
3. choose the next free ID in the _4–_99 range
4. ensure missionId string, filename, and final execute call all use the same ID

Blind / no-file-access workflow (MANDATORY):
If the AI cannot directly inspect the folder contents, it must NOT guess that _4 is free.
It must first ask the user whether the first community slot already exists.

Required behavior:
- first ask whether the correct category prefix _4 already exists in the exact target folder
- supply must ask for MissionSupply_4
- intel must ask for MissionIntel_4
- disrupt must ask for MissionDisrupt_4
- if _4 does not exist, use _4
- if _4 already exists, ask for the used IDs or the folder contents
- only after that, choose the next free _4–_99 ID
- never overwrite an existing file
- never assume an ID is free without confirmation when folder inspection is unavailable

Examples:
- if MissionSupply_4 and MissionSupply_5 already exist, do not overwrite them
- choose MissionSupply_6 if 6 is the next free ID
- if the AI cannot inspect the folder itself, it must first ask:
  "Does MissionSupply_4 already exist in Missionframework/SideMissions/land/supply/easy/?"

4. Gold standard reference
Community missions must mirror the official gold-standard structure used by files like:

Missionframework/SideMissions/land/supply/easy/MissionSupply_1.sqf

The AI should copy the structure and field order, but use a new community ID.

5. Exact function contract (MANDATORY)
Use this exact parameter format:

params [
    ["_slotData", [], [[]]],
    ["_caller", objNull, [objNull]]
];

Do not rename:
- _slotData
- _caller

Use this exact final execution call pattern:

[_slotData, _caller, "<category>", "<difficulty>", <numericId>, _missionDefinition] call MWF_fnc_executeMissionTemplate;

The category string, difficulty string, numeric ID, filename, and missionId field must all match.

Use this exact mission metadata style and keep the fields in this order:

- missionId
- title
- description
- category
- difficulty
- allowedZoneTypes
- allowUndercover
- usesOpfor
- usesBluforSupport
- usesCivilians
- usesRebels
- compositionKey

- enemyTierSource

- rebelTierSource

- rewardThreatUndercover

- minRebelRep
- failIfRepTooLow

Do not invent new metadata keys unless the framework is explicitly updated to read them.

7. Approved current land zone types
For current land side mission authoring, use only zone types already supported by the current placement model:

- town
- military
- factory
- capital

Do not invent or assume support for:
- industrial
- roadside
- rural
- coastal
- airport
- harbor
- custom tags not already implemented

If a mission idea needs a special environment, express it through the description, sceneVariant, and compositionKey rather than unsupported zone tags.

8. Zone discipline rule (MANDATORY)
Use the narrowest approved zone selection that correctly fits the mission concept.

Do NOT broaden allowedZoneTypes for variety, flavor, fallback, or creativity.
Do NOT add an extra approved zone type unless the mission concept explicitly requires it.

Examples:
- if town + factory is sufficient, use only town + factory
- do not add military just because it also sounds plausible
- do not add capital unless the mission concept specifically needs capital-scale placement

Community missions should stay within the runtime-supported modern land sceneVariant style.

Intel examples:

Disrupt examples:

If a new sceneVariant is proposed, treat it as a request for framework/runtime support first, not as something the mission file can force by itself.

10. Strict minimalism rule (MANDATORY)
Community mission templates must use the smallest correct implementation that fits the mission concept.

The AI must NOT add extra flavor structure, broaden scope, or add optional metadata behavior unless it is explicitly required by the mission idea or explicitly supported by the framework.

Prefer:
- the smallest valid zone set
- the simplest valid sceneVariant
- the most conservative runtime assumptions
- the minimum necessary explanation in comments and assetRules

Do not "improve" the mission by adding extra allowed zones, extra behavior claims, or extra author comments.

11. Mod agnostic rules
Mission templates must remain mod agnostic.

Do NOT hardcode:
- faction-specific unit classnames
- vehicle classnames
- prop classnames
- weapon classnames
- item classnames
- era-specific asset classnames

Mission templates describe intent only.
Runtime asset selection must come from presets and progression systems.

12. Preset / scaling source rules
OPFOR
- source: active OPFOR preset
- scaled by: world tier

BLUFOR
- source: active BLUFOR preset
- scaled by: player base tier

Civilians
- source: active civilian preset

Rebels
- source: active resistance preset
- scaled by: rebel tier

13. CompositionKey rules
compositionKey is allowed as a future composition hook and naming signal.
However:
- community mission templates must still work if the composition is not populated yet
- compositionKey must not be treated as permission to hardcode props in the mission file
- runtime remains map agnostic first

14. Undercover / civilians / rebels rules
Current normal land patterns:
- supply: usually allowUndercover = true
- intel: usually allowUndercover = true
- disrupt: usually allowUndercover = false unless there is a strong stealth justification

Current normal usage:
- supply often usesCivilians = true
- intel often usesCivilians = true
- disrupt usually usesCivilians = false
- usesRebels should remain false unless rebel-specific logic is intentionally designed

Do not mark every mission as stealth-capable by default.
Do not mark every mission as rebel-enabled by default.

15. AssetRules discipline rule (MANDATORY)
assetRules may describe intent, constraints, and allowed framework-level expectations only.

assetRules must NOT:
- promise specific runtime behavior unless explicitly guaranteed by the framework
- claim that a specific objective carrier, spawn method, or interaction already exists unless confirmed
- describe speculative implementation as if it is already supported

Good:
- "Use active OPFOR preset infantry scaled by world tier for site security."
- "Use sceneVariant to guide runtime presentation."

Bad:
- "Officer carries the intel objective."
- "Van is spawned through convoy logic."
- "Runtime guarantees a vehicle delivery phase."

16. Reward rules
Rewards are fixed per mission.
Do not randomize rewards.

Guidance:
- supply should lean heavier on Supplies
- intel should lean heavier on Intel
- disrupt should carry stronger Threat impact than the other two

rewardThreatUndercover should usually be:
- 0 for clean stealth-friendly supply/intel work
- low but non-zero when a mission still creates visible disruption

17. Difficulty rules
Allowed difficulty labels:
- easy
- medium
- hard

Difficulty should scale through:
- guardCount
- patrolRadius
- clearRadius
- addOfficer
- reward tuning

Do not scale difficulty by hardcoding faction assets in the template.

18. Comment header discipline rule (MANDATORY)
Use only the approved header structure shown in the gold-standard skeleton.

Do not add extra header fields such as:
- Description:
- Notes:
- Runtime:
- Special Rules:

The header should remain minimal and standardized for diff readability.

19. Community safety boundaries
Community AI mission templates in _4–_99 must NOT:
- edit core framework logic
- edit presets
- edit economy systems
- edit progression systems
- edit tier/threat formulas
- edit persistence
- overwrite official/internal missions
- author main operations through this side mission pipeline

20. Gold-standard authoring workflow (MANDATORY)
1. choose land + category + difficulty
2. inspect the exact target folder
3. if the folder cannot be inspected directly, ask whether the correct category-specific _4 file already exists
4. if _4 is taken, ask for the used IDs or folder contents
5. find the next free _4–_99 ID
6. mirror the official gold-standard file structure
7. use the exact params contract
8. use the exact missionDefinition field order
9. use only approved zone types
10. use the narrowest valid zone set
11. use no hardcoded classnames
12. keep the mission mod agnostic
13. avoid promising unsupported runtime behavior
14. set fixed rewards
15. save with matching filename / missionId / numeric call ID

21. Final self-check (MANDATORY)
Before finalizing the file, the AI must verify all of the following:

1. Did I ask about _4 first if folder visibility was unavailable?
2. Did I use the correct path, prefix, and ID?
3. Did I keep the exact params contract?
4. Did I keep the exact field order?
5. Did I use only approved zone types?
6. Did I use the narrowest valid zone set?
7. Did I avoid inventing metadata keys?
8. Did I avoid promising unsupported runtime behavior?
9. Did I avoid hardcoded classnames?
10. Did I keep the file minimal and gold-standard compliant?
