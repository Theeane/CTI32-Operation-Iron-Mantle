AI Mission / Operation Generation Instructions (_4–_99)

Purpose
These instructions define how future LLM-generated content must be authored for Operation Iron Mantle.
Official/internal missions _1–_3 are reserved in every side-mission domain/type/difficulty folder and must never be overwritten.
Community-created content belongs in the _4–_99 range only.

1. Scope Split
There are two different authoring targets:
- Side Missions
- Main Operations

Do not mix their structure.
Side missions live under Missionframework/SideMissions/.
Main operations live under Missionframework/MainOp/.

2. Side Mission Folder Structure (MANDATORY)
All side missions must be domain-aware.
Valid domains:
- land
- naval
- air

Valid side mission types:
- supply
- intel
- disrupt

Valid difficulties:
- easy
- medium
- hard

Side mission path format:
- Missionframework/SideMissions/{domain}/{type}/{difficulty}/Mission<TypePascal>_<id>.sqf

Examples:
- Missionframework/SideMissions/land/supply/easy/MissionSupply_4.sqf
- Missionframework/SideMissions/naval/intel/medium/MissionIntel_7.sqf
- Missionframework/SideMissions/air/disrupt/hard/MissionDisrupt_12.sqf

3. Reserved Range Rules (MANDATORY)
Official/internal baseline files _1–_3 are reserved in every side mission folder.
Examples:
- MissionSupply_1..3
- MissionIntel_1..3
- MissionDisrupt_1..3

Community-generated side missions must use _4–_99 only.
Never rename, overwrite, or duplicate _1–_3.

4. Side Mission ID Workflow (MANDATORY)
Before creating a new side mission, the AI must first inspect the exact target folder.
It must:
1. scan the folder contents
2. identify all used numeric suffixes
3. select the next free ID in the _4–_99 range
4. avoid duplicates

Do not assume the next ID without scanning the actual folder.
Do not scan the wrong domain/type/difficulty folder.

5. Side Mission Definition Contract (MANDATORY)
Each side mission file must remain mod agnostic and must define structured mission metadata.
Required fields:
- missionId
- title
- description
- category
- difficulty
- allowedZoneTypes
- allowUndercover
- usesOpfor
- usesBluforSupport
- usesCivilians
- usesRebels
- compositionKey
- enemyTierSource
- bluforTierSource
- rebelTierSource
- rewardSupplies
- rewardIntel
- rewardThreat
- rewardTier
- rewardThreatUndercover
- requiresRebelCooperation
- minCivilianRep
- minRebelRep
- failIfRepTooLow
- notes
- assetRules

Optional fields may include:
- requiresRoad
- requiresSettlement
- requiresMilitary
- requiresCoast
- minDistanceFromBase
- maxDistanceFromBase
- fallbackSupplies
- fallbackIntel

6. Side Mission Domain Rules
The mission file must match the selected domain.
Examples:
- land missions should be authored for normal ground gameplay
- naval missions must assume naval-safe anchor usage and maritime-compatible gameplay
- air missions must assume air-capable flow and future air marker support

Do not place a naval mission in a land folder.
Do not place an air mission in a naval folder.

7. Side Mission Reward Rules
Rewards must be fixed per mission file.
Do not randomize reward values.
Undercover completions should use rewardThreatUndercover when relevant.
If fallbackSupplies/fallbackIntel are defined, they must be fixed values.

8. Side Mission Variety Rules
When generating new side missions, vary:
- domain
- type
- difficulty
- allowedZoneTypes
- civilian/rebel use
- stealth vs loud compatibility
- composition usage

Avoid generating structurally identical missions repeatedly.
Each mission needs a unique short identity so UI metadata does not feel duplicated.

9. No Hardcoded Classnames (MANDATORY)
Mission files must remain mod agnostic.
Do NOT hardcode:
- OPFOR unit classnames
- BLUFOR unit classnames
- civilian classnames
- rebel classnames
- vehicle classnames
- aircraft classnames
- boat classnames
- prop/object classnames

All runtime asset selection must come from active presets and framework systems.

10. Asset Source Rules
Mission files describe intent only.
Runtime asset selection comes from presets and progression systems.

OPFOR
- source: preset/opfor
- scaled by: world tier / mission systems

BLUFOR
- source: preset/blufor
- scaled by: player base tier

Civilians
- source: preset/civilians

Rebels
- source: preset/resistance
- scaled by: rebel tier / reputation rules

11. Composition Rules
All layout, props, and scene dressing must use composition keys.
Do not hardcode world objects directly into mission definitions.
Mission files may reference composition keys only.

12. Main Operations Scope
Main operations are not stored in SideMissions.
They live in Missionframework/MainOp/ and must use the existing operation naming pattern:
- MWF_fn_op_<name>.sqf

Main operations are strategic operations, not generic side missions.
They must be authored to fit one of the approved operation effect models:
- tier reduction
- tier block
- preset reduction

13. Main Operation Metadata Expectations
When adding or editing a main operation, the AI must provide or preserve:
- unique operation key
- display title
- short description for UI panel use
- effect type / effect summary
- fallback summary
- cooldown-aware behavior compatibility

Main operations must be written so they can be described cleanly in UI with:
- title
- short description
- effect/reward text
- fallback text
- status / cooldown state

14. Tier / Fallback Rules for Operations
Main operations must respect the current campaign rules:
- tier reduction always attempts to reduce tier
- after 50 percent map control, Tier 3 is the permanent floor
- if reduction cannot go below Tier 3, fallback Supplies/Intel must be used instead
- tier block operations can still apply after 50 percent map control
- tier block currently targets a 60 minute grace period
- if a tier block is already active, a new block effect must convert to fallback Supplies/Intel instead of stacking

15. Domain-Aware Player Requests
When a player asks for a mission, the AI must identify whether they want:
- land
- naval
- air

The AI must also identify:
- side mission or main operation
- supply / intel / disrupt if side mission
- easy / medium / hard if side mission

Do not assume land by default if the request clearly asks for naval or air.
If the user does not specify a domain, preserve the requested framework rules or ask/inspect context before choosing.

16. Community Safety Rules
Community-generated content in the _4–_99 range must NOT:
- modify presets directly
- modify economy systems directly
- modify tier/threat framework logic directly
- modify core mission registry logic directly
- overwrite official/internal _1–_3 files

17. Authoring Workflow (MANDATORY)
Side missions:
1. identify domain
2. identify type
3. identify difficulty
4. inspect the exact target folder
5. find the next free _4–_99 ID
6. create a mod-agnostic mission file using the side mission metadata contract
7. keep rewards fixed
8. use composition keys only
9. ensure the mission fits its selected domain

Main operations:
1. identify the intended operation effect model
2. preserve mod-agnostic preset behavior
3. provide unique title/description/effect/fallback language
4. ensure the operation fits the central cooldown and fallback rules
5. save using the existing MainOp naming pattern

18. Final Rule
Never output generic content that ignores domain.
Every generated mission or operation must clearly match the current Operation Iron Mantle framework structure.
